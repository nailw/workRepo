## 8. 重试后invalidDate仍被Date.MAX覆盖

### 用户

2026-07-21 18:08、18:11和18:13连续重试，错误仍然是：

```text

production_date='2026-07-21 +08'
ERROR: date out of range: "169087565-03-15 +08"
```

完整错误标识包括：

```text
projectCode: DMHR
sceneCode: HR
method: BusinessDataManager.uploadMonthlyInventory(Context;InputStream;)
requestType: POST
level: ERROR
threadName: SceneSerThr-P:DMHR-S:HR-39
```

### 助手

判断Excel解析已经不是问题，实际待写入对象的`invalidDate`仍为`Date.MAX`。建议在最终`insertOrUpdateBatch(ktMonthlyInventories)`之前强制归一化：

```java
if (!ktMonthlyInventories.isEmpty()) {
    ktMonthlyInventories.forEach{ inventory ->
        if (inventory.expireDate != null && inventory.expireDate.isFinite()) {
            inventory.invalidDate = inventory.expireDate
        }
        if (inventory.productionDate == null || !inventory.productionDate.isFinite()) {
            inventory.productionDate = inventory.inboundDate
        }

        log.info(
            "KT库存写入前，uniqueId：" + inventory.uniqueId +
            "，productionDate：" + inventory.productionDate +
            "，invalidDate：" + inventory.invalidDate +
            "，inboundDate：" + inventory.inboundDate +
            "，expireDate：" + inventory.expireDate
        )
    }

    dh.insertOrUpdateBatch(ktMonthlyInventories)
}
```

如果写入前日志已经显示`invalidDate=3000-01-01`，但SQL仍为`169087565-03-15`，则需要检查`Interface33_ImportMonthlyInventory.invalidDate`节点字段配置：

- 是否存在`Date.MAX`默认值；
- 是否存在保存前计算公式；
- 是否是只读或计算属性；
- 是否在模型保存阶段覆盖代码赋值。

7. 下一步应查看“KT库存写入前”日志，并检查`Interface33_ImportMonthlyInventory.invalidDate`字段默认值和计算规则。
