# 备份与归档规范

## 1. 目的

本规范用于确保项目聊天记录、开发决策、开发文档和开发规范持续保存到 GitHub，并可通过提交历史追溯。

## 2. 必须归档的内容

以下内容属于强制备份范围：

- 与项目有关的聊天记录；
- 需求、方案、关键决策、变更原因和执行结果；
- 开发文档、接口文档、数据结构说明和测试记录；
- 开发规范、命名规范、发布规范和运维说明；
- 重要脚本、配置模板及问题排查记录。

## 3. 目录约定

- `聊天记录/YYYY-MM-DD.md`：按日期保存聊天原文、可恢复内容和决策摘要；
- `docs/`：保存开发文档、方案、测试记录和文档索引；
- `开发规范/`：保存长期适用的编码与开发规范；
- 各业务目录：保存与具体月份或项目直接相关的成果文件。

## 4. 归档时机

- 每次完成一项工作后，更新当天聊天记录；
- 产生关键决定、需求变化或方案调整时，立即补充决策记录；
- 新建或修改重要文档、规范后，在同一批提交中上传；
- 提交前检查文档中的相对链接是否有效。

## 5. 聊天记录格式

每份聊天记录至少包含：

1. 日期与时区；
2. 用户原始要求；
3. 助手处理摘要；
4. 已确认的决策；
5. 修改或新增的文件；
6. 尚未完成、存在风险或需要确认的事项。

能够取得完整原文时应保存完整原文；只能取得部分内容时，必须明确标记“部分记录”或“根据可见上下文整理”，不得冒充完整逐字记录。

## 6. 安全要求

当前 `nailw/workRepo` 是公开仓库。以下内容不得直接写入公开仓库：

- 密码、访问令牌、API Key、Cookie、私钥；
- 未脱敏的个人信息、客户信息或公司机密；
- 生产数据库连接信息；
- 其他不适合公开传播的内容。

发现上述内容时，聊天记录中只保留脱敏摘要，并标明已脱敏。需要保存完整敏感原文时，应改用私有仓库。

## 7. 完整性检查

每次归档应确认：

- 文件已经存在于 GitHub 分支中；
- 关键文档已被索引页引用；
- 文件名和目录名称清晰、稳定；
- 不存在失效链接；
- 提交信息能够说明本次备份内容。
