# 核料流程小白详解

- 日期：2026-07-15
- 适用对象：第一次接触排程核料逻辑的开发和业务人员
- 说明范围：依据当前提供的 MaterialManager、MaterialDemand、MaterialSupply、MaterialFulfill 等代码整理

## 1. 什么是核料

这里的“核料”不是质量检验，而是生产排程中的物料齐套检查。

它要回答的是：

1. 当前要生产什么产品、生产多少；
2. 根据 BOM，需要哪些原料、半成品和包材；
3. 每种物料实际需要多少；
4. 库存、采购、在制品等供应是否足够；
5. 这些供应是否满足批次、有效期、供应商、车间等约束；
6. 主物料不足时能否使用替代料；
7. 配套物料能否按同一个比例齐套；
8. 最早什么时候能够把物料准备齐；
9. 物料准备时间是否会推迟生产步骤。

核料最终生成 `MaterialFulfill`，表示某条供应给某条需求分配了多少数量。

## 2. 核心对象

| 对象 | 通俗解释 | 示例 |
|---|---|---|
| `WorkOrder` | 生产工单 | 生产产品 A 1,000 瓶 |
| `Piece` | 工单拆分后的生产批次 | 第一批生产 500 瓶 |
| `WorkOrderStep` | 工艺路线中的步骤定义 | 配料、灌装、包装 |
| `PieceStep` | 某批次实际执行的步骤 | 第一批的灌装步骤 |
| `InputOnStep` | 一个步骤需要投入什么 | 灌装需要半成品和瓶子 |
| `OutputOnStep` | 一个步骤会产出什么 | 包装后产出成品 A |
| `MaterialDemand` | 根据产量算出的实际物料需求 | 需要原料 X 100kg |
| `MaterialSupply` | 可以提供物料的供应节点 | 3410 库位有 X 60kg |
| `MaterialFulfill` | 供应和需求的分配结果 | 3410 库位给该需求 60kg |

对象关系：

```text
WorkOrder
  → Piece
    → PieceStep
      → InputOnStep
        → MaterialDemand

库存、采购、生产计划
  → MaterialSupply

MaterialDemand + MaterialSupply
  → MaterialFulfill
```

## 3. 核料总流程

主流程可以概括为：

```text
检查核料开关
  ↓
生成步骤投入和产出
  ↓
同步全部物料供应
  ↓
扣除其他车间占用
  ↓
生成实际物料需求
  ↓
删除未冻结分配并重占冻结分配
  ↓
重新进行供需分配
  ↓
计算物料最早可用时间
  ↓
把分配结果发送给 DM
```

主代码：

```java
if (!this.enabledCheckInputMaterial()) {
    return
}

this.synDemandAndSupply()
this.clearUnfrozenAndReOccupyFrozenMaterialFulfills()

var pieceSteps = Group<PieceStep>().toList()
this.assignSuppliesToMaterialDemands()
this.updateFulfillTimeToPieceStepTime(pieceSteps)
this.sendFulfillsToDM()
```

## 4. 检查核料开关

主要方法开头都会执行：

```java
if (!this.enabledCheckInputMaterial()) {
    return
}
```

场景没有开启核料时，后续需求、供应和分配都不会生成。

## 5. 生成步骤投入和产出

系统首先要知道每个生产步骤需要什么，以及步骤结束后会产出什么。

```java
this.synInputAndOutputOnStep()
```

系统根据场景参数选择工序 BOM 或普通 BOM：

```java
var pattern = this.owner.sceneParameter?.orderAndProcessPattern ?? ""

if (pattern == ModelConstants.Pattern23) {
    this.synInputAndOutOnStepByOperationBOM()
} else {
    this.synInputAndOutOnStepByBOM()
}
```

### 5.1 工序 BOM

工序 BOM 会明确物料应该在哪一道工序投入。例如：

| 工序 | 投入 | 产出 |
|---|---|---|
| 配料 | 原料 X、原料 Y | 中间液体 |
| 灌装 | 中间液体、瓶子 | 已灌装产品 |
| 包装 | 已灌装产品、纸盒 | 成品 A |

系统通过 DM 接口取得工序 BOM，在对应步骤创建 `InputOnStep`，并在最后步骤创建 `OutputOnStep`。

### 5.2 普通 BOM

普通 BOM 中，有 `KTInputConstraint` 特殊步骤约束的物料会被放到指定步骤；其余普通 BOM 物料默认放到每个 Piece 的第一步骤。

每个 Piece 的最后选中步骤会创建产出。这个产出还能继续成为后续生产任务的半成品供应。

## 6. 同步物料供应

系统先删除旧供应，再从多个来源重新生成统一供应池：

```java
this.synInventoryToMaterialSupply()
this.synIngredientToMaterialSupply()
this.synPurchaseOrderToMaterialSupply()
this.synPurchaseRequestToMaterialSupply()
this.synOtherScenesProductionPlanToMaterialSupply()
this.synOutputStepToMaterialSupply()
```

### 6.1 ERP库存

普通原辅包库存拆成：

1. 已经可以使用的 `quantity`；
2. 还在待检的 `detectQuantity`。

待检库存的预计释放时间为：

```text
入库时间 + 产品检验周期
```

半成品库存只使用可用量，不再拆待检量，并保留半成品生产日期、入库日期和虚拟可用量等信息。

### 6.2 原料药到货计划

系统通过 DM 接口取得原料药供应计划并转换为 `MaterialSupply`。

### 6.3 采购订单和采购计划

采购数量大于零时会生成供应。采购物料通常也需要检验，因此预计释放时间是到货时间加检验周期。

### 6.4 其他场景生产计划

其他车间或其他排程场景未来会生产的半成品，也可以作为当前场景未来可用的供应。

### 6.5 本车间生产计划

当前车间前一个任务步骤的 `OutputOnStep` 会转换为供应，供后续任务使用。

## 7. 扣除其他车间占用

ERP账面库存不一定都能被当前场景使用。

例如：

```text
库存总量：100kg
其他车间已占用：30kg
当前场景真正可用：70kg
```

系统通过 `synMaterialSupplyAvailableQuantity()` 更新真正可用的供应量。

## 8. 生成 MaterialDemand

系统遍历所有可以核料的 `PieceStep`，读取步骤的 `InputOnStep`，生成实际需求。

```java
Group<PieceStep>().forEach{ pieceStep ->
    if (pieceStep.cannotCheckInputReason != "") {
        return
    }

    pieceStep.workOrderStep?.inputsOnStep?.forEach{ input ->
        var obj = new MaterialDemand()
        MaterialDemand.updateAttributeByPieceStepAndInput(obj, pieceStep, input)
        toAdd.add(obj)
    }
}
```

### 8.1 需求量计算

例如生产 1,000 瓶，每瓶消耗原料 X 0.1kg：

```text
初始需求 = 1,000 × 0.1 = 100kg
```

如果收率为 95%：

```text
实际需求 = 100 ÷ 0.95 = 105.263kg
损耗数量 = 105.263 - 100 = 5.263kg
```

代码对应：

```java
obj.initDemandQuantity = input.isFirstPriority
    ? input.inputFactor * pieceStep.quantity
    : 0.0

obj.demandQuantity = obj.initDemandQuantity / scrapRate
obj.scrapQuantity = obj.demandQuantity - obj.initDemandQuantity
```

### 8.2 产品代码和物料代码

```java
obj.productId = pieceStep.productId
obj.materialId = input.materialId
```

- `productId` 是当前要生产的成品；
- `materialId` 是生产该成品所需要的原料或半成品。

## 9. 处理冻结分配

重新核料前，系统删除所有未冻结的旧分配。冻结分配表示业务人员已经确认某需求必须继续使用某条供应。

例如需求 100kg，已经冻结 30kg：

```text
已满足：30kg
还需要重新分配：70kg
```

系统通过 `demandUniqueId` 和 `supplyUniqueId` 找回需求和供应，然后增加需求满足量并扣减供应剩余量。

## 10. 正式分配

系统一次性读取全部需求和供应，并先选择第一优先级需求：

```java
var supplyNode = MaterialSupply.queryAllData(context())
var demandNode = MaterialDemand.queryAllData(context())
var firstDemands = MaterialDemand.queryFirstPriorityFromInputDemand(demandNode)
```

需求按照交期和步骤顺序排序。通常越早需要生产的任务越先占用供应。

## 11. 普通需求和替代料

没有配套组的需求进入 `assignSuppliesToFirstReplaceDemand()`。

系统先尝试主物料。主物料不足时，如果存在 `replaceGroupId`，就按 `replacePriority` 依次尝试替代料。

例如需求 100kg：

```text
主物料 X：60kg
第一替代料 Y：30kg
第二替代料 Z：20kg
```

分配结果可以是：

```text
X分配60kg
Y分配30kg
Z分配10kg
合计100kg
```

## 12. 配套组

配套组表示几种物料需要按同一个满足比例一起准备。

例如：

```text
瓶子需要100个，可满足100%
瓶盖需要100个，只能满足80%
标签需要100张，可满足90%
```

配套组最大满足率取最小值，即 80%。正式分配时统一按 80%处理：

```text
瓶子分配80个
瓶盖分配80个
标签分配80张
```

系统会先临时模拟每种物料的满足率，再取瓶颈比例正式分配。如果不允许混用配套组且不能完全满足，这一整套可能不分配。

## 13. 供应过滤

供应首先按物料代码匹配：

```text
MaterialSupply.productId == MaterialDemand.materialId
```

物料代码相同后，还要经过多种约束。

### 13.1 数量

剩余数量不足的供应会被排除。半成品、国际订单半成品可能使用专门的虚拟数量或可用量算法。

### 13.2 设备组禁用物料

任务已经排到具体设备后，系统根据设备组查询禁止使用的半成品，并排除对应供应。第一次核料尚未排产时不执行这一规则。

### 13.3 国际订单半成品

国际成品使用 BOM 半成品时，还会按照生产月份等条件过滤。

### 13.4 不可混批

需求 100kg，两个库存批次分别为 60kg、40kg：

- 允许混批：可以合计满足100kg；
- 禁止混批：两个批次都不能单独满足，因此不能使用。

### 13.5 有效期

如果库存会在任务需求日期前过期，就不能使用。

### 13.6 指定供应商和上游车间

开启对应约束后，只允许使用指定供应商或指定上游车间的供应。

## 14. MaterialFulfill

每一次实际分配都会生成 `MaterialFulfill`。

例如：

```text
需求：任务A需要原料X 100kg
供应1：库存批次B001，分配60kg
供应2：采购订单P001，分配40kg
```

需求的已满足量增加，供应的剩余量减少。

## 15. 计算物料最早可用时间

一个步骤通常需要多种物料，必须等所有物料都准备好才能开工。

| 物料 | 最早可用时间 |
|---|---|
| 原料 X | 7月15日 |
| 原料 Y | 7月17日 |
| 包材 Z | 7月16日 |

该步骤的物料齐套时间是最晚的 7 月 17 日。系统再用这个时间更新 `PieceStep` 的计划时间。

## 16. 发送给 DM

最后执行 `sendFulfillsToDM()`，把需求使用了哪些供应、分配数量、可用时间和缺料情况发送给 DM。

## 17. 禁用库位需求如何进入流程

需求是：某些产品核料时不能使用指定 ERP 库位。

正确判断关系：

```text
MaterialDemand.productId
  → 查询 KTProductRouting.NoLocationId

MaterialSupply.sapStockingPointId
  → 当前库存供应所在库位
```

例如产品 12020228 禁用 3411：

| 物料 | 库位 | 数量 | 结果 |
|---|---|---:|---|
| RM0001 | 3410 | 60 | 可用 |
| RM0001 | 3411 | 50 | 禁用 |
| RM0001 | 3412 | 40 | 可用 |

不能在库存同步阶段把 3411 全局删除，因为其他产品仍可能允许使用 3411。正确位置是公共供应查询 `MaterialSupply.queryAvailableFromInputSupplies()`。

## 18. 一个完整例子

计划生产产品 12020228 共 1,000 件。

BOM：

```text
原料X：每件0.1kg
包材Y：每件1个
```

原料 X 收率为 95%，实际需求：

```text
1,000 × 0.1 ÷ 0.95 = 105.263kg
```

供应：

| 物料 | 来源 | 库位 | 数量 | 时间 |
|---|---|---|---:|---|
| 原料X | 可用库存 | 3410 | 60kg | 7月15日 |
| 原料X | 可用库存 | 3411 | 50kg | 7月15日 |
| 原料X | 待检库存 | 3412 | 40kg | 7月18日 |
| 包材Y | 库存 | — | 800个 | 7月15日 |
| 包材Y | 采购订单 | — | 300个 | 7月17日 |

如果产品 12020228 禁用 3411，原料 X 只能使用 3410 的 60kg 和 3412 的 40kg，合计 100kg，仍缺 5.263kg。

包材 Y 使用库存 800 个和采购 200 个，能够完全满足，但要等采购在 7 月 17 日可用。

最终结果：

```text
原料X：缺5.263kg
包材Y：完全满足，7月17日齐套
整个任务仍处于缺料状态
```

## 19. 一句话总结

核料不是简单判断总库存够不够，而是：

> 针对具体生产任务，在满足替代料、配套组、批次、有效期、供应商、车间、设备组和库位等约束的前提下，确定哪些供应可以使用、分别使用多少、是否缺料，以及最早什么时候能够开工。
