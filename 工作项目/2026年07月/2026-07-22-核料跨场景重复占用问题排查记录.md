# 核料跨场景重复占用问题排查记录

- 日期：2026-07-22
- 涉及成品：`09020004`
- 涉及半成品：`12030005`
- 当前状态：根因已基本定位，尚未修改代码
- 影响范围：相同订单数据同时存在于多个规划场景时的跨场景核料占用

## 1. 问题现象

成品`09020004`对半成品`12030005`的需求量为`240000`，当前场景显示满足量为0，但DM满足情况中能够看到5条本车间生产计划供应：

| 供应编号 | 分配数量 |
|---|---:|
| BCP2-108-1 | 51000 |
| BCP2-108-2 | 51000 |
| BCP2-108-3 | 51000 |
| BCP2-108-4 | 51000 |
| BCP2-108-5 | 36000 |
| 合计 | 240000 |

5条供应合计正好等于当前需求量。多个规划场景使用相同订单数据，相关分配在页面上均显示供应给`CS02`，但CS02并不存在多份独立的实际需求。

## 2. 相关代码链路

### 2.1 本场景生产计划生成供应

`MaterialSupply.updateAttributeByPieceStepAndOutput`中的供应唯一标识：

```java
obj.fromSceneId = context.sceneCode()
obj.workShopId = pieceStep.resourceGroups.firstOrNull()?.workShopId ?? ""
obj.productId = output.productId
obj.uniqueId = pieceStep.pieceStepId + output.productId
```

`uniqueId`只包含`pieceStepId+output.productId`，不包含项目代码或规划场景代码。

### 2.2 创建物料分配

`MaterialFulfill.createFulfillByDemandAndSupply`复制供需信息：

```java
fulfill.supplyUniqueId = supply.uniqueId
fulfill.demandWorkshopId = demand.workshopId
fulfill.supplyWorkShopId = supply.workShopId
```

需求车间来自：

```java
obj.workshopId = pieceStep.resourceGroups.firstOrNull()?.workShopId ?? ""
```

因此页面显示的`CS02`是订单/资源所属车间代码，不等同于生成分配记录的规划场景代码。

### 2.3 DM记录保留规划场景

`DMFulfill.updateAttributeFromPP`会保存规划场景和项目：

```java
var ppSceneId = Utils.anyToString(map["ppSceneId"])
var ppProjectId = Utils.anyToString(map["ppProjectId"])
obj.ppSceneId = ppSceneId
obj.ppProjectId = ppProjectId
obj.uniqueId = Utils.anyToString(map["uniqueId"]) + ppSceneId + ppProjectId
```

所以`ppSceneId`并未在DMFulfill转换时丢失。DMFulfill通过追加场景和项目避免记录主键互相覆盖，但供应占用计算仍只使用`supplyUniqueId`。

### 2.4 跨场景占用计算

`synMaterialSupplyAvailableQuantity`读取DM中所有场景的满足记录，移除当前场景后按`supplyUniqueId`汇总：

```java
data.removeIf{
    Utils.anyToString(it["ppProjectId"]) == context().projectCode() &&
    Utils.anyToString(it["ppSceneId"]) == context().sceneCode()
}
var fulfillMap = data.groupBy{ Utils.anyToString(it["supplyUniqueId"]) }
supplies.forEach{ supply ->
    var fulfills = fulfillMap.getOrDefault(supply.uniqueId,new List<Map>())
    if (fulfills.isNotEmpty()) {
        supply.occupiedByOtherWorkshop = fulfills.sumByDouble{
            Utils.anyToDouble(it["quantity"])
        }
        supply.availableQuantity = maxOf(
            0.0,
            supply.supplyQuantity-supply.occupiedByOtherWorkshop
        )
        supply.leftQuantity = supply.availableQuantity.roundHalfDown(7)
    }
}
```

这里没有判断其他规划场景中的记录是否只是同一订单、同一需求的场景副本。

## 3. 根因分析

多个规划场景使用相同订单数据时，很可能生成相同的：

```text
pieceStepId
output.productId
MaterialSupply.uniqueId
MaterialFulfill.supplyUniqueId
```

随后形成以下链路：

```text
场景A对12030005分配240000
→ 分配结果同步DM
→ 场景B、场景C读取DM数据
→ 相同supplyUniqueId被识别为其他场景的真实占用
→ occupiedByOtherWorkshop增加240000
→ availableQuantity和leftQuantity变为0
→ 当前场景需求显示未满足
```

本质上是：相同业务订单在不同规划方案中的副本，被当成多个需要同时执行的真实需求，导致物理供应被重复占用。

## 4. CS02字段说明

页面上的“该物料供应给的场景代码=CS02”大概率对应`demandWorkshopId`，其来源是资源组的`workShopId`。

它表示业务车间，不表示该记录来自哪个规划场景。真正用于区分规划场景的是：

```text
ppProjectId
ppSceneId
```

因此多个规划场景中都显示CS02并不能证明记录属于同一个规划场景。

## 5. 订单编号是否能避免问题

当前代码并不使用订单编号判断供应是否相同，而是使用：

```java
obj.uniqueId = pieceStep.pieceStepId + output.productId
```

所以判断标准是：

- 不同订单编号产生不同`pieceStepId`和`supplyUniqueId`：通常不会发生本次碰撞；
- 不同订单编号仍产生相同`pieceStepId`和`supplyUniqueId`：仍会发生；
- 相同订单编号但不同场景产生不同`supplyUniqueId`：不会按同一供应汇总。

订单编号不重复只能降低出现概率，不能从代码层面保证问题不发生。

## 6. 验证方法

对三个规划场景中`12030005`相关记录同时检查：

```text
ppProjectId
ppSceneId
demandWorkshopId
workOrderNr
demandUniqueId
pieceStepId
supplyUniqueId
quantity
```

如果不同`ppSceneId`下出现相同的`workOrderNr+demandUniqueId+supplyUniqueId`，即可确认是相同订单的场景副本被重复计入。

同时检查供应节点：

```text
uniqueId
supplyQuantity
occupiedByOtherWorkshop
availableQuantity
leftQuantity
leftQuantityVritual
fromSceneId
workShopId
```

## 7. 修复方向

修复前必须先明确：多个场景是同一车间的互斥排程方案，还是需要同时执行的真实生产场景。

如果属于互斥排程方案，跨场景占用同步需要识别相同业务需求的场景副本，不能简单累计。可考虑基于以下业务字段去重或排除：

```text
workOrderNr
demandUniqueId
demandPieceStepId
supplyUniqueId
demandWorkshopId
ppProjectId
ppSceneId
```

具体组合需要结合场景业务关系确定。

## 8. 不建议的直接修改

不建议直接把`ppSceneId`拼入`MaterialSupply.uniqueId`作为最终修复：

```java
obj.uniqueId = context.sceneCode() + "|" + pieceStep.pieceStepId + "|" + output.productId
```

这样虽然能避免场景间ID碰撞，但也会让每个场景都拥有一份独立的物理供应，可能导致同一批240000被多个场景分别使用，形成超分配。

## 9. 当前结论

问题不在12030005的供应总量，也不是DMFulfill丢失了规划场景字段。当前最可能的直接原因是：

```text
相同订单存在于多个规划场景
+ 供应唯一ID未包含可区分业务副本的范围
+ 跨场景占用仅按supplyUniqueId累加
= 相同需求在不同规划场景中被重复计为真实占用
```

在业务规则确认前暂不修改核料分配算法。
