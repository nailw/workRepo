# 核料本车间生产供应不分配排查记录

- 整理日期：2026-07-16
- 项目：ASJW京卫作业排程
- 场景：气雾剂车间场景
- 关联业务：物料核料、半成品供应、国内/国际订单分类
- 涉及成品：`09020003`
- 涉及半成品：`12030001`
- 供应示例：`BCP1-108-1`
- 文档性质：聊天排查记录与后续开发依据

## 1. 问题现象

物料需求界面存在成品`09020003`对半成品`12030001`的需求，需求量为`240000`。物料供应界面可以查询到多条来自本场景生产计划的供应，例如：

| 供应编号 | 物料代码 | 来源 | 可见数量 |
|---|---|---|---:|
| BCP1-108-1 | 12030001 | 来自本场景生产计划 | 51000 |
| BCP1-108-2 | 12030001 | 来自本场景生产计划 | 51000 |
| BCP1-108-3 | 12030001 | 来自本场景生产计划 | 51000 |
| BCP1-108-4 | 12030001 | 来自本场景生产计划 | 51000 |
| BCP1-108-5 | 12030001 | 来自本场景生产计划 | 36000 |

供应对象的`leftQuantityVritual`有值，但核料后没有生成对应的`MaterialFulfill`，界面中所有供应均未被选择。

## 2. 相关核料主流程

本次问题位于以下核料链路：

1. 同步步骤投料和产出；
2. 同步库存、采购、其他场景生产计划、本场景生产计划等供应；
3. 根据`InputOnStep`生成`MaterialDemand`；
4. 删除未冻结分配并重占冻结占用；
5. 查询可用供应；
6. 按需求和供应生成`MaterialFulfill`；
7. 根据分配更新物料最早可用时间；
8. 将分配结果发送给DM。

核心调用关系：

```java
var supplies=MaterialSupply.queryAvailableFromInputSupplies(supplyNode,firstDemand,Constant.EPS,context)
var fulfillTuple=MaterialDemand.assignAvailableSuppliesToDemand(context,supplies,firstDemand,firstDemand,false,tempFlag,hasUserFulfillRatio,totalRemainRatio)
```

## 3. 供应来源和有效数量

本场景生产计划供应通过以下逻辑生成：

```java
Group<PieceStep>().forEach{ pieceStep->
    pieceStep.workOrderStep?.outputsOnStep?.forEach{ output->
        var obj=new MaterialSupply()
        MaterialSupply.updateAttributeByPieceStepAndOutput(context(),obj,pieceStep,output)
        MaterialSupply.updateSupplySortKey(obj)
        toAdd.add(obj)
    }
}
```

国际半成品专用取量方法：

```java
static Double getGuojiBanchengpingAvailableQty(MaterialSupply supply){
    if(MaterialSupply.isProductionPlanSupply(supply)){
        return supply.leftQuantityVritual??0.0
    }
    return supply.leftQuantity??0.0
}
```

生产计划供应包括本场景和其他场景：

```java
return supply.dataSource==ModelConstants.FromOtherSceneProductionPlan||supply.dataSource==ModelConstants.FromThisSceneProductionPlan
```

因此，本场景生产供应的有效数量设计上主要使用历史字段`leftQuantityVritual`。该字段名是模型真实字段，必须保持该拼写，不能改成`leftQuantityVirtual`。

## 5. 日志确定的删除位置

完整日志中同一轮查询的顺序：

```text
21:27:03.575  [核料供应-初始]
21:27:03.578  [核料供应-设备组过滤后] count=5
21:27:03.579  [核料供应-数量过滤前]
21:27:03.579  [核料供应-数量过滤后] count=0
```

结论：

> 5条供应是在数量过滤阶段一次性全部删除的。

随后内部方法再次对空集合查询，所以出现第二组：

```text
初始供应=0
设备组过滤后=0
数量过滤后=0
```

这不是第二个独立问题，而是第一次数量过滤返回空集合后的连锁结果。

## 6. 最终根因

数量过滤代码：

```java
if(isGuojiChengpingUseBomBanchengping){
    outputSupplies.removeIf{ MaterialSupply.getGuojiBanchengpingAvailableQty(it)<=Constant.EPS }
}else if(isDomesticSemiDemand){
    outputSupplies.removeIf{ it.leftQuantityVritual<=Constant.EPS }
}else{
    outputSupplies.removeIf{ it.leftQuantity<leastQty }
}
```

实际分类结果：

```text
isGuojiChengpingUseBomBanchengping=false
isDomesticSemiDemand=false
```

所以程序错误进入普通物料分支：

```java
outputSupplies.removeIf{ it.leftQuantity<leastQty }
```

本场景生产计划供应应按`leftQuantityVritual`参与半成品核料，但当前分类失败后读取了普通供应字段`leftQuantity`，导致供应全部被删除。

### 6.1 国际成品分类失败

国际成品集合：

```java
return KT<KTProductRouting>().filter{
    (it?.productType1??"")=="C"&&(it?.customType??"")=="国际订单"
}.map{ it.productId }.distinct()
```

成品`09020003`实际维护：

```text
productType1=C
customType=空
productFrom108=true
```

由于`customType`为空，`09020003`不在`guojiChengpingIds`中。

### 6.2 国内半成品分类失败

国内半成品集合：

```java
var banchengpingIds=context.KT<KTProductRouting>().filter{
    (it?.productType1??"")=="B"&&it.customType=="国内订单"
}.map{ it.productId }
```

半成品`12030001`实际维护：

```text
productType1=B
customType=空
productFrom108=true
```

由于`customType`为空，`12030001`不在国内`banchengpingIds`中。

最终两个判断均失败：

```text
国际成品+BOM半成品=false
国内半成品=false
```

这就是本场景生产供应不分配的直接原因。

## 7. 当前建议处理方案

### 7.1 当前订单为国内订单

维护基础数据：

```text
09020003.customType=国内订单
12030001.productType1=B
12030001.customType=国内订单
```

按照当前代码，决定`isDomesticSemiDemand`的关键是`12030001`的`productType1=B`和`customType=国内订单`。

保存后需要重新计算相关属性或重新初始化场景，使`PiecePlanner.banchengpingIds`重新生成，再执行核料。

### 7.2 当前订单为国际订单

维护：

```text
09020003.productType1=C
09020003.customType=国际订单
```

同时确认：

```text
PiecePlanner.bomBanchengpingIds包含12030001
```

两个条件同时成立后，`isGuojiChengpingUseBomBanchengping`才会返回`true`。

### 7.3 108线产品是否等同国际订单

当前`productFrom108=true`不参与国内或国际类型判断。只有业务明确规定“所有108线C类产品均使用国际半成品规则”时，才考虑：

```java
return KT<KTProductRouting>().filter{
    (it?.productType1??"")=="C"&&((it?.customType??"")=="国际订单"||(it?.productFrom108??false))
}.map{ it.productId }.distinct()
```

若国内订单也可能在108线生产，则不能这样修改。

## 8. 代码设计风险

### 8.1 国内和国际分类标准不一致

当前：

- 国际规则根据需求成品的订单类型判断；
- 国内规则根据投料半成品自身的`customType`判断。

如果同一个半成品可以同时供应国内和国际订单，给半成品维护固定订单类型存在歧义。长期建议统一为：

```text
需求成品customType=国内订单 + 投料属于BOM半成品
→ 国内半成品规则

需求成品customType=国际订单 + 投料属于BOM半成品
→ 国际半成品规则
```

### 8.2 不能只修改数量过滤

如果只把查询阶段改为读取`leftQuantityVritual`，后续普通分配仍会执行：

```java
var qty=minOf(remainQty,supply.leftQuantity).roundHalfUp(6)
```

供应即使通过查询，也可能继续得到零分配。因此应修复分类，使需求进入完整半成品分配分支；或者统一修改查询、分配和扣减三处取量规则。

### 8.3 国内半成品分配的剩余量风险

现有代码：

```java
var qty1=minOf(remainQty,supply.leftQuantityVritual).roundHalfUp(6)
var qty=remainQty.roundHalfUp(6)
remainQty=0.0
var fulfillRatio=demandQuantity>0.0?qty/demandQuantity:0.0
```

若需求量为240000、第一条供应只有51000，实际`MaterialFulfill.quantity`可能为51000，但`remainQty`直接变为0，满足率按240000计算为100%。基础数据修复后，应验证是否只选择第一条供应就停止；若与业务不符，需要继续修复此段逻辑。

### 8.4 BOM半成品推导算法待验证

`querySemiProductIdsByBom()`当前可能使用：

```java
!bomProductIds.contain(rawMaterialId)
```

若业务对半成品的定义是“既是上级BOM子项，同时自己也有BOM”，则条件可能应为：

```java
bomProductIds.contain(rawMaterialId)
```

该方法被库存、BOM投料等多个模块调用，修改前必须用实际BOM数据验证影响范围。

## 9. DO日志调试经验

用户提供的既有代码确认标准日志语法：

```java
log.info("下发失败，原因："+e.toString())
```

但当前`MaterialSupply`方法中直接写`log.info(String)`曾出现名称解析报错，而直接`log(String)`能够输出INFO日志。说明日志对象或函数可能受所属类和方法上下文影响。编写时应优先参考当前类中已成功编译的现有写法，必要时封装到`Utils`统一调用。

本次实践证明，复杂过滤链应一次性打印以下内容，不应逐层反复加断点：

- 产品和物料代码；
- 当前需求或批次标识；
- 每个过滤阶段前后的供应数量；
- 关键分类布尔值；
- 每条候选供应的来源、`leftQuantity`和`leftQuantityVritual`。

问题确认后应删除临时高频日志，避免影响核料性能。
