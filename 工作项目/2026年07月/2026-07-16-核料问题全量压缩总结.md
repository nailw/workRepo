# 核料问题全量压缩总结

- 整理日期：2026-07-16
- 项目：ASJW京卫作业排程
- 范围：核料流程、供应分配、国内/国际半成品分类、替代料满足、DO日志调试
- 相关对象：WorkOrder、Piece、PieceStep、InputOnStep、MaterialDemand、MaterialSupply、MaterialFulfill
- 相关产品：09020003
- 相关物料：12030001

## 1. 核料总体流程

核料主流程可以压缩为：

```text
核料开关判断
→ 同步步骤投料和产出
→ 同步库存、采购、其他场景生产计划、本车间生产计划供应
→ 根据InputOnStep生成MaterialDemand
→ 删除未冻结分配并重占冻结分配
→ 查询可用供应
→ 按优先级、替代组和配套组分配供应
→ 生成MaterialFulfill
→ 更新需求、供应剩余量
→ 计算PieceStep最早可用时间
→ 向DM发送分配结果
```

核心分配调用：

```java
var supplies=MaterialSupply.queryAvailableFromInputSupplies(supplyNode,firstDemand,Constant.EPS,context)
var fulfillTuple=MaterialDemand.assignAvailableSuppliesToDemand(context,supplies,firstDemand,firstDemand,false,tempFlag,hasUserFulfillRatio,totalRemainRatio)
```

## 2. 供应和需求的关键字段

### MaterialDemand

- `demandQuantity`：需求量；
- `fulfillQuantity`：已满足数量；
- `leftRatio`：第一优先级或替代组剩余满足比例；
- `isFirstPriority`：是否第一优先级；
- `isFirstReplace`：是否替代组第一顺位；
- `replaceGroupId`：替代组；
- `replacePriority`：替代顺序；
- `inputFactor`：投料系数；
- `pieceStepQuantity`：步骤批量。

需求量生成：

```java
obj.initDemandQuantity=input.isFirstPriority?input.inputFactor*pieceStep.quantity:0.0
obj.demandQuantity=obj.initDemandQuantity/scrapRate
obj.leftRatio=input.isFirstPriority?1.0:0.0
```

因此，替代料需求通常会出现：

```text
替代料demandQuantity=0
替代料leftRatio=0
```

这不代表替代料不能被使用，而是表示替代料不单独形成原始需求量，只有在主料不足时参与满足。

### MaterialSupply

- `leftQuantity`：普通供应剩余量；
- `leftQuantityVritual`：模型中的历史字段拼写，半成品/生产计划供应使用的虚拟剩余量；
- `availableReleaseTime`：供应可用时间；
- `dataSource`：库存、采购、其他场景生产、本场景生产等来源；
- `uniqueId`：供应唯一标识。

必须使用模型真实字段：

```java
leftQuantityVritual
```

不能自行改写为`leftQuantityVirtual`。

## 3. 本车间供应不分配问题

### 3.1 现象

成品`09020003`需要物料`12030001`，供应包括：

```text
BCP1-108-1
BCP1-108-2
BCP1-108-3
BCP1-108-4
BCP1-108-5
```

供应来自本场景生产计划，界面可见虚拟量，但没有生成MaterialFulfill。

### 3.2 已排除原因

- 订单未暂停；
- 替代有效期没有提前返回；
- 需求量不是0；
- `firstDemand.leftRatio=1.0`；
- 物料代码匹配成功；
- Piece和Peizhi步骤存在；
- 生产月份过滤未执行；
- `forbiddenMix=false`；
- 设备组禁用物料代码为空。

### 3.3 日志定位

日志显示：

```text
初始供应 count=5
设备组过滤后 count=5
数量过滤前
数量过滤后 count=0
```

因此5条供应是在数量过滤阶段被全部删除的。

### 3.4 根因

数量过滤代码：

```java
if(isGuojiChengpingUseBomBanchengping){
    outputSupplies.removeIf{ MaterialSupply.getGuojiBanchengpingAvailableQty(it)<=Constant.EPS }
}else if(isDomesticSemiDemand){
    outputSupplies.removeIf{ it.leftQuantityVritual<=Constant.EPS }
}else{
    outputSupplies.removeIf{ it.leftQuantity<leastQty }
}
```

实际分类：

```text
isGuojiChengpingUseBomBanchengping=false
isDomesticSemiDemand=false
```

所以程序把生产计划半成品当作普通物料，读取`leftQuantity`，而不是`leftQuantityVritual`，最终供应被删除。

## 4. 国内/国际订单分类规则

### 国际订单判断

```java
(it?.productType1??"")=="C"&&(it?.customType??"")=="国际订单"
```

成品`09020003`当时配置：

```text
productType1=C
customType=空
productFrom108=true
```

因此没有进入国际成品集合。

### 国内半成品判断

```java
var banchengpingIds=context.KT<KTProductRouting>().filter{
    (it?.productType1??"")=="B"&&it.customType=="国内订单"
}.map{ it.productId }
var isDomesticSemiDemand=banchengpingIds.contain(materialDemand.materialId)
```

半成品`12030001`当时配置：

```text
productType1=B
customType=空
productFrom108=true
```

因此也没有进入国内半成品集合。

### 国内订单的正确维护

如果业务上是国内订单：

```text
09020003.customType=国内订单
12030001.productType1=B
12030001.customType=国内订单
```

其中按照当前代码，真正决定`isDomesticSemiDemand`的是半成品`12030001`的类型和客户类型。

### 国际订单的正确维护

如果业务上是国际订单：

```text
09020003.productType1=C
09020003.customType=国际订单
```

同时确认：

```text
bomBanchengpingIds包含12030001
```

108线产品字段当前不会自动代表国内或国际订单，除非明确修改代码规则。

## 5. 替代料满足规则

### 5.1 替代料需求量

主物料：

```text
isFirstPriority=true
demandQuantity=正常需求量
leftRatio=1.0
```

替代料：

```text
isFirstPriority=false
demandQuantity=0
leftRatio=0
```

所以使用替代料时：

- 主物料的`demandQuantity`不会被改成0；
- 替代料自身的原始需求量通常是0；
- 替代料实际满足数量写入替代料需求的`fulfillQuantity`；
- 主替代组是否满足，通过第一优先级需求的`leftRatio`判断。

### 5.2 替代料满足数量

替代分配调用：

```java
MaterialFulfill.createFulfillByDemandAndSupply(demand,supply,qty)
```

这里的`demand`是实际使用的替代料需求，不是主需求。

因此：

```text
MaterialFulfill.quantity=实际替代料分配量
替代料MaterialDemand.fulfillQuantity+=实际分配量
主料MaterialDemand.fulfillQuantity通常不增加
主料leftRatio按满足比例扣减
```

### 5.3 如何判断使用了替代料

通过`MaterialFulfill.demandUniqueId`找到实际需求，再找到同一替代组的第一顺位需求：

```java
var actualDemand=MaterialDemand.findByUniqueId(context,fulfill.demandUniqueId)
var firstDemand=MaterialDemand.findFirstReplacePriorityByMaterialDemand(context,actualDemand)
var isReplace=actualDemand!=null&&firstDemand!=null&&actualDemand.uniqueId!=firstDemand.uniqueId
```

也可以辅助判断：

```text
actualDemand.replaceGroupId不为空
actualDemand.isFirstReplace=false
actualDemand.replacePriority大于第一顺位
```

最准确的是比较实际需求和第一顺位需求的`uniqueId`。

### 5.4 物料满足情况展示什么代码

当前`MaterialFulfill`是按实际需求创建的：

```java
createFulfillByDemandAndSupply(demand,supply,qty)
```

因此满足情况中的实际物料代码应展示替代料代码。

推荐页面同时展示：

| 字段 | 含义 |
|---|---|
| 主需求物料代码 | 第一顺位主物料 |
| 实际满足物料代码 | 实际分配的物料，包括替代料 |
| 是否替代料 | 实际需求是否不是第一顺位 |
| 替代组 | `replaceGroupId` |
| 替代顺位 | `replacePriority` |
| 主需求数量 | 主物料原始需求量 |
| 本次满足数量 | `MaterialFulfill.quantity` |
| 实际供应编号 | `supplyUniqueId`或`nr` |

截图中的数据含义：

```text
主料08020144：需求183.6，满足0
替代料08020003：需求0，满足183.6
绑定供应物料：08020003
```

这表示替代料`08020003`实际满足了主料`08020144`的183.6需求。

## 6. 当前发现的代码风险

### 6.1 国内半成品分配可能提前结束

现有逻辑：

```java
var qty1=minOf(remainQty,supply.leftQuantityVritual).roundHalfUp(6)
var qty=remainQty.roundHalfUp(6)
remainQty=0.0
```

如果需求240000、首条供应51000，实际只生成51000满足量，却把剩余需求直接置为0，可能只使用第一条供应就结束。

需要确认这是否符合国内半成品“整批通过”的业务规则。如果不符合，应使用实际分配量扣减：

```java
remainQty=remainQty-qty1
```

而不是直接使用总需求量归零。

### 6.2 国内和国际判断标准不一致

当前国际根据成品订单类型判断，国内根据半成品自身客户类型判断。如果同一半成品同时服务国内和国际订单，建议统一改为：

```text
成品订单类型 + 投料是否属于BOM半成品
```

不要只依赖半成品自身的客户类型。

### 6.3 BOM半成品推导条件需要验证

`querySemiProductIdsByBom()`中的`!bomProductIds.contain(rawMaterialId)`是否符合业务定义需要用实际BOM验证。若半成品必须同时拥有自己的BOM，条件可能需要改为`bomProductIds.contain(rawMaterialId)`。

## 7. DO语法和调试经验

已确认：

- 方法签名由平台界面配置；
- 使用`var`、`new List<T>()`、`Group<T>()`、`context.KT<T>()`；
- 空值使用`?.`和`??`；
- 集合包含使用`contain`；
- 闭包使用`filter{ it... }`、`forEach{ item -> ... }`；
- 支持`for(Int i=1;i<13;i+=1)`；
- 日期支持`month()`、`year()`、`dayOfMonth()`、`minusMonths()`、`minusDays()`；
- 模型字段必须按实际字段名使用，例如`leftQuantityVritual`；
- DO日志一般可使用`log.info(String)`，但具体模型上下文可能存在名称解析差异，当前`MaterialSupply`方法应优先参考已成功编译的日志写法；
- 复杂过滤排查应一次性打印产品、物料、数量、分类条件和每阶段集合数量，确认后删除高频临时日志；
- 代码减少无意义换行，保持适合DO编辑器复制。

## 8. 最终结论

本次本车间供应不分配的直接链路是：

```text
09020003和12030001客户类型未维护
→ 国际分类失败
→ 国内半成品分类也失败
→ 本场景生产计划供应被按普通物料处理
→ 使用leftQuantity而不是leftQuantityVritual
→ 数量过滤阶段5条供应全部删除
→ 分配方法收到空供应
→ 不生成MaterialFulfill
```

替代料满足的核心规则是：

```text
主料保留原始需求量
替代料原始需求量通常为0
实际满足量写入替代料需求和MaterialFulfill
替代组整体满足程度通过主需求leftRatio判断
物料满足情况应同时展示主需求物料和实际使用物料
```
