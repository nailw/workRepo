# 开发文档-JWAS-产品禁用库位核料改造

- 文档日期：2026-07-15
- 适用模块：JWAS 车间排程 / 核料
- 需求状态：方案设计
- 目标版本：待定
- 禅道任务：待补充

## 一、🧭 禅道任务

| 项目 | 内容 |
|---|---|
| 任务名称 | 产品禁用库存库位核料改造 |
| 禅道任务地址 | 待补充 |
| 业务背景 | 车间排程通过 BOM 生成物料需求。部分半成品订单在核料时不能使用指定库存代码中的物料。 |
| 需求目标 | 在 `KTProductRouting` 维护产品禁用库位；核料时按需求成品代码过滤库存供应，禁止命中库位参与普通物料、替代料及配套组分配。 |
| 影响范围 | 产品路径基础数据、库存供应转换、公共供应过滤、冻结分配重新占用、核料测试。 |

### 1.1 需求示例

ERP库存清单：

| 库位代码 | 库存名称 | 状态 | 是否激素/高活物料库位 |
|---|---|---|---|
| 3410 | 原料药三车间材料库 | 有效 | 否 |
| 3411 | 原料药一车间材料库 | 有效 | 是 |
| 3412 | 原料药二车间材料库 | 有效 | 否 |

产品配置和核料结果：

| 产品代码 | 禁用库位 | 核料时可使用库位 |
|---|---|---|
| 12020228 | 3411 | 3410、3412 |
| 12020071 | 空 | 3410、3411、3412 |

### 1.2 核心原则

禁用库位不能在库存同步时全局删除。3411 对产品 12020228 禁用，并不表示它对其他产品也禁用。规则必须在“某条供应是否可以满足某条需求”时按需求产品动态判断。

## 二、🪶 输入

### 2.1 输入对象

| 输入对象 | 来源 | 关键内容 |
|---|---|---|
| `KTProductRouting` | 产品路径基础数据 | 成品 `productId`、禁用库位 `NoLocationId` |
| `InventoryIn` | ERP库存接口 | 库存物料、批次、数量、`sapStockingPointId`、库位名称 |
| `MaterialDemand` | 核料需求节点 | 成品 `productId`、需求物料 `materialId`、需求量、替代组、配套组 |
| `MaterialSupply` | 核料供应节点 | 供应物料 `productId`、供应来源、剩余量、库存库位 |
| `MaterialFulfill` | 已有核料结果 | 需求唯一ID、供应唯一ID、冻结标记、分配数量 |
| `SceneParameter` | 场景参数 | 核料开关、配套组是否可混用等 |

### 2.2 关键字段

| 对象 | 字段 | 类型 | 含义/约束 |
|---|---|---|---|
| `KTProductRouting` | `productId` | String | 当前配置对应的成品代码 |
| `KTProductRouting` | `NoLocationId` | String | 禁用库存代码；建议支持逗号分隔多个值 |
| `InventoryIn` | `sapStockingPointId` | String | ERP库存代码，当前模型已存在 |
| `MaterialDemand` | `productId` | String | 当前要生产的成品代码 |
| `MaterialDemand` | `materialId` | String | 当前需求的原料或半成品代码 |
| `MaterialSupply` | `productId` | String | 当前供应节点的物料代码 |
| `MaterialSupply` | `sapStockingPointId` | String | 供应节点所在库存代码，需要新增 |
| `MaterialSupply` | `dataSource` | String | 区分库存、采购、生产计划等供应来源 |
| `MaterialFulfill` | `demandUniqueId` | String | 分配对应的需求唯一ID |
| `MaterialFulfill` | `supplyUniqueId` | String | 分配对应的供应唯一ID |
| `MaterialFulfill` | `frozenFlag` | Boolean | 是否为冻结分配 |

### 2.3 字段关系

```text
MaterialDemand.productId → KTProductRouting.productId → NoLocationId
MaterialDemand.materialId → MaterialSupply.productId
MaterialSupply.sapStockingPointId → 与 NoLocationId 比较
```

`MaterialDemand.productId` 是要生产的成品，`MaterialDemand.materialId` 是 BOM 需要的物料，两者不能混用。

## 三、✉️ 输出

| 输出 | 内容 |
|---|---|
| 产品禁用库位配置 | 产品路径表可以维护一个或多个禁用库存代码 |
| 带库位的库存供应 | 可用库存、待检库存、半成品库存转为 `MaterialSupply` 后保留库位代码 |
| 新分配结果 | 普通物料、替代料、配套组分配均不使用命中禁用库位的库存供应 |
| 冻结分配处理 | 冻结供应命中禁用库位时不再重占，解除或删除后重新核料 |
| 其他来源供应 | 采购订单、采购计划、生产计划等无实际库位供应继续正常参与核料 |
| 核料齐套时间 | 基于过滤后的合法供应重新计算物料最早可用时间 |

## 四、🌋 操作流程

### 4.1 现有核料主流程

```text
检查核料开关
  → 同步步骤投入和产出
  → 同步库存、采购、生产计划等供应
  → 扣除其他车间占用
  → 根据步骤投入生成物料需求
  → 删除未冻结分配并重占冻结分配
  → 普通需求、替代料、配套组供需分配
  → 计算物料最早可用时间
  → 发送核料结果给 DM
```

主调用：

```java
if (!this.enabledCheckInputMaterial()) return
this.synDemandAndSupply()
this.clearUnfrozenAndReOccupyFrozenMaterialFulfills()
var pieceSteps = Group<PieceStep>().toList()
this.assignSuppliesToMaterialDemands()
this.updateFulfillTimeToPieceStepTime(pieceSteps)
this.sendFulfillsToDM()
```

### 4.2 需求生成

系统根据 `PieceStep` 和 `InputOnStep` 生成 `MaterialDemand`。关键赋值：

```java
obj.productId = pieceStep.productId
obj.materialId = input.materialId
obj.demandQuantity = obj.initDemandQuantity / scrapRate
```

其中 `productId` 可直接用于查询产品路径的禁用库位。

### 4.3 供应生成

供应来源包括：

1. ERP可用库存；
2. ERP待检库存；
3. ERP半成品库存；
4. 原料药到货计划；
5. 采购订单和采购计划；
6. 其他场景生产计划；
7. 本车间步骤产出。

本需求只过滤具有实际 ERP 库位的库存供应，不过滤采购及生产计划供应。

### 4.4 新分配入口

普通需求、替代料和配套组最终都会进入：

```text
MaterialSupply.queryAvailableFromInputSupplies()
```

因此禁用库位过滤放在该公共方法中，可以覆盖：

- 第一优先级物料；
- 下一替代顺位物料；
- 配套组临时满足率计算；
- 配套组正式分配。

### 4.5 冻结分配入口

冻结分配通过 `reOccupyFrozenMaterialFulfill()` 直接重占需求和供应，不经过公共供应查询。因此冻结分配必须单独校验，否则产品新增禁用库位后，历史冻结结果仍会继续使用非法库位。

## 五、💡 设计逻辑

### 5.1 模型改造

#### KTProductRouting

新增字段：

```text
NoLocationId String
```

建议约定：

- 空字符串表示不限制；
- 单个库位示例：`3411`；
- 多个库位示例：`3411,3413`；
- 同一产品多条路径记录时，合并所有禁用库位并去重。

#### MaterialSupply

新增字段：

```text
sapStockingPointId String
```

`MaterialDemand` 已保存成品代码，不需要为本需求重复增加产品字段。

### 5.2 库存供应保留库位

以下三个转换方法都要增加库位赋值：

1. `updateAttributeByInventoryAvailableNode`；
2. `updateAttributeByInventoryDetectNode`；
3. `updateAttributeBySemiInventoryNode`。

关键代码统一为：

```java
obj.plantId = inventory.plantId
obj.sapStockingPointId = inventory.sapStockingPointId
obj.productId = inventory.productId
obj.productName = inventory.productName
```

### 5.3 查询产品禁用库位

建议在 `MaterialSupply` 类新增静态方法：

```text
queryNoLocationIdsByProductId(Context context, String productId) → List<String>
```

参考实现：

```java
var result = new List<String>()
if (productId == "") return result
context.KT<KTProductRouting>().filter{ it.productId == productId }.forEach{ routing ->
    Utils.splitStringToStringList(routing.NoLocationId ?? "").forEach{ id ->
        if (id != "" && !result.contain(id)) result.add(id)
    }
}
return result
```

如果平台模型字段实际创建为 `noLocationId`，代码必须同步使用小写字段名。

### 5.4 公共供应过滤

修改 `MaterialSupply.queryAvailableFromInputSupplies()`，在按需求物料取得候选供应后立即过滤库位：

```java
var supplyMap = inputSupplies.groupBy{ it.productId }
var outputSupplies = supplyMap.getOrDefault(materialDemand.materialId, new List<MaterialSupply>())
var noLocationIds = MaterialSupply.queryNoLocationIdsByProductId(conetxt, materialDemand.productId)
if (!noLocationIds.isEmpty()) outputSupplies.removeIf{ MaterialSupply.isInventorySupply(it) && (it.sapStockingPointId ?? "") != "" && noLocationIds.contain(it.sapStockingPointId) }
```

之后继续执行原有规则：

1. 设备组禁用物料；
2. 国内/国际半成品可用量；
3. 生产月份；
4. 不可混批；
5. 有效期；
6. 指定供应商；
7. 指定上游车间。

### 5.5 冻结分配处理

建议规则：禁用库位优先级高于冻结。冻结分配命中禁用库位时，不增加需求满足量、不扣减供应数量，将该冻结分配删除后由本轮重新分配。

冻结循环中的处理顺序应改为：

```text
查找需求
  → 查找供应
  → 供应不存在：记录为无效冻结
  → 查询成品禁用库位
  → 库位命中：记录为无效冻结
  → 供应合法：更新需求满足量和供应剩余量
  → 批量删除无效冻结分配
```

参考核心判断：

```java
var supply = supplyMap.getOrDefault(fulfill.supplyUniqueId, new List<MaterialSupply>()).firstOrNull()
if (supply == null) { invalidFrozenFulfills.add(fulfill); return }
var noLocationIds = MaterialSupply.queryNoLocationIdsByProductId(context, demand.productId)
if (MaterialSupply.isInventorySupply(supply) && noLocationIds.contain(supply.sapStockingPointId ?? "")) { invalidFrozenFulfills.add(fulfill); return }
```

当前代码在确认供应存在前就执行 `demand.fulfillQuantity += fulfill.quantity`。此次修改应一并调整顺序，避免供应已经消失但需求仍被计算为已满足。

### 5.6 MaterialFulfill 是否增加库位

本需求不强制给 `MaterialFulfill` 增加库位字段，因为可以通过 `supplyUniqueId` 找回当前 `MaterialSupply`。如果 DM 输出或业务审计需要直接查看实际使用库位，可选增加：

```text
sapStockingPointId String
```

创建 Fulfill 时从供应复制。

### 5.7 性能设计

第一版可直接从 `context.KT<KTProductRouting>()` 查询。由于配套组会进行临时计算和正式分配，同一需求可能重复查询产品路径。

如果核料耗时明显增加，可选择：

1. 给 `MaterialDemand` 增加 `noLocationId`，生成需求时提前写入；或
2. 核料开始时构建 `Map<productId, List<locationId>>` 并向下传递。

建议先完成正确性验证，再依据运行日志决定是否缓存。

### 5.8 不建议的实现

以下方式不可采用：

- 在 `synInventoryToMaterialSupply()` 中把禁用库位全局删除；
- 只过滤第一优先级物料而遗漏替代料；
- 只改普通需求而遗漏配套组临时分配；
- 只处理新分配而保留非法冻结分配；
- 对采购、生产计划等无实际库位供应执行库位排除。

## 六、🚩 功能设计补充

| 功能点 | 注意点 | 建议处理 |
|---|---|---|
| 多库位配置 | 文本中可能存在空格、重复值 | 使用统一拆分工具并去重 |
| 多产品路径 | 同一产品可能有多条 `KTProductRouting` | 合并全部禁用库位；若业务要求按当前路径，需要增加路径定位条件 |
| 空库位 | 历史供应或非库存供应库位为空 | 不因空库位直接排除 |
| 待检库存 | 虽未释放但仍属于具体库位 | 同样执行禁用过滤 |
| 半成品库存 | 通过库存生成，具有具体库位 | 同样执行禁用过滤 |
| 冻结供应消失 | 当前代码仍可能增加需求满足量 | 先校验供应，再更新需求 |
| 日志 | 过滤后出现缺料时难以定位 | 建议输出产品、物料、库位、供应ID的调试日志 |

## 附录B：验收标准

1. 未配置禁用库位时，核料结果与改造前一致；
2. 配置禁用库位后，命中库位的库存供应不会生成新的 `MaterialFulfill`；
3. 普通、替代、配套组分配行为一致；
4. 非库存供应不受影响；
5. 非目标产品仍可使用同一库位；
6. 冻结分配不会绕过禁用规则；
7. 过滤结果会正确影响缺料状态和最早齐套时间；
8. 性能没有出现不可接受的下降；
9. 所有回归用例通过后方可进入顾问测试。
