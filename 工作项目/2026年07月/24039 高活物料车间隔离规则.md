# 四、新增查询产品禁用库位的方法

建议在 `MaterialSupply` 类新增静态方法。

方法配置：

```text
方法名：queryNoLocationIdsByProductId
所属类：MaterialSupply
返回值：List<String>
```

参数：

| 参数名       | 类型      |
| --------- | ------- |
| context   | Context |
| productId | String  |

方法体：

```java
var result = new List<String>()
if (productId == "") {
    return result
}

context.KT<KTProductRouting>().filter{ it.productId == productId }.forEach{ routing ->
    var locationIds = Utils.splitStringToStringList(routing.NoLocationId ?? "")
    locationIds.forEach{ locationId ->
        if (locationId != "" && !result.contain(locationId)) {
            result.add(locationId)
        }
    }
}

return result
```

这个方法实现了：

1. 根据成品代码查找 `KTProductRouting`；
2. 读取 `NoLocationId`；
3. 支持多个库位；
4. 自动去重；
5. 没有配置时返回空列表；
6. 同一产品存在多条路径时，暂时合并全部禁用库位。

例如：

```text
产品12020228
第一条路径：3411
第二条路径：3411,3413
```

最终结果是：

```text
3411、3413
```

# 五、修改公共供应查询

需要修改：

```text
MaterialSupply.queryAvailableFromInputSupplies
```

现有方法开头：

```java
var supplyMap = inputSupplies.groupBy { it.productId }
var outputSupplies = supplyMap.getOrDefault(materialDemand.materialId, new List<MaterialSupply>())
```

在这两行后面立即加入：

```java
// 根据当前生产的成品代码查询禁用库存代码
var noLocationIds = MaterialSupply.queryNoLocationIdsByProductId(conetxt, materialDemand.productId)

// 只过滤ERP库存供应，采购和生产计划不受影响
if (!noLocationIds.isEmpty()) {
    outputSupplies.removeIf{
        MaterialSupply.isInventorySupply(it) &&
        (it.sapStockingPointId ?? "") != "" &&
        noLocationIds.contain(it.sapStockingPointId)
    }
}
```

修改后的方法开头：

```java
var supplyMap = inputSupplies.groupBy { it.productId }
var outputSupplies = supplyMap.getOrDefault(materialDemand.materialId, new List<MaterialSupply>())

// 产品禁用库存代码过滤
var noLocationIds = MaterialSupply.queryNoLocationIdsByProductId(conetxt, materialDemand.productId)
if (!noLocationIds.isEmpty()) {
    outputSupplies.removeIf{
        MaterialSupply.isInventorySupply(it) &&
        (it.sapStockingPointId ?? "") != "" &&
        noLocationIds.contain(it.sapStockingPointId)
    }
}

// 原有设备组禁用物料逻辑
var productDemand = conetxt.KT<KTSingleResourceSpeedExtend>().filter{ it.productId == materialDemand.productId }
if (!productDemand.isEmpty()) {
    var scheduledPieceSteps = conetxt.Group<PieceStep>().filter{
        it.productId == materialDemand.productId &&
        it.pieceId == materialDemand.pieceId &&
        it.isScheduled == true
    }

    if (!scheduledPieceSteps.isEmpty()) {
        scheduledPieceSteps.forEach{ p ->
            var usedResourceGroupId = conetxt.Group<Resource>().filter{
                it.resourceId == p.usedResourceId
            }.firstOrNull()?.resourceGroupId ?? ""

            if (!usedResourceGroupId.isEmpty()) {
                var materialsID = productDemand.filter{
                    it.resourceGroupId == usedResourceGroupId
                }.firstOrNull()?.NoMaterialsID ?? ""

                if (!materialsID.isEmpty()) {
                    outputSupplies.removeIf{ it.productId == materialsID }
                }
            }
        }
    }
}
```

这里继续沿用你原方法的参数名：

```java
conetxt
```

虽然这个单词拼写应该是 `context`，但当前方法参数如果就叫 `conetxt`，新增代码也必须使用 `conetxt`，否则会找不到变量。

# 六、为什么使用 materialDemand.productId

这里必须写：

```java
materialDemand.productId
```

不能写：

```java
materialDemand.materialId
```

因为：

```text
materialDemand.productId
= 当前正在生产的成品代码
= 12020228
```

而：

```text
materialDemand.materialId
= 当前BOM需要的物料代码
= RM0001
```

禅道任务要求的是：

> 根据正在生产的产品代码查询禁用库存代码。

所以正确关系是：

```text
12020228
→ 查询KTProductRouting
→ 得到NoLocationId=3411
→ 排除sapStockingPointId=3411的库存供应
```

---

# 七、为什么只过滤库存供应

过滤条件中必须保留：

```java
MaterialSupply.isInventorySupply(it)
```

因为 `MaterialSupply` 不只有 ERP 库存，还包括：

* 采购订单；
* 采购计划；
* 原料药到货计划；
* 其他场景生产计划；
* 本车间步骤产出。

这些供应没有实际库存代码。

正确逻辑：

```text
ERP库存 + 库位命中禁用列表
→ 排除

采购/生产计划 + 库位为空
→ 继续参与核料
```

# 十、冻结分配怎么处理

建议先完成前面的必改部分并验证，再处理冻结分配。

原因是冻结分配走：

```java
reOccupyFrozenMaterialFulfill()
```

不会调用：

```java
queryAvailableFromInputSupplies()
```

因此存在以下场景：

```text
原来冻结使用3411
后来给产品配置NoLocationId=3411
重新核料时仍继续占用3411
```

如果禅道任务要求“任何情况下都不能使用禁用库位”，冻结逻辑也要改。

## 1. 增加无效冻结列表

在方法开始处增加：

```java
var invalidFrozenFulfills = new List<MaterialFulfill>()
```

## 2. 在更新需求前先检查供应

你现在的代码是先执行：

```java
demand.fulfillQuantity += fulfill.quantity
```

然后再查询供应。

应该调整为先查供应：

```java
var noLocationIds = MaterialSupply.queryNoLocationIdsByProductId(context, demand.productId)

fulfills.forEach{ fulfill ->
    var supply = supplyMap.getOrDefault(fulfill.supplyUniqueId, new List<MaterialSupply>()).firstOrNull()

    if (supply == null) {
        invalidFrozenFulfills.add(fulfill)
        return
    }

    if (
        MaterialSupply.isInventorySupply(supply) &&
        noLocationIds.contain(supply.sapStockingPointId ?? "")
    ) {
        invalidFrozenFulfills.add(fulfill)
        return
    }

    demand.fulfillQuantity += fulfill.quantity

    if (firstPriorityDemand != null) {
        var demandQty = demand.pieceStepQuantity * demand.inputFactor
        var ratio = demandQty > 0.0 ? fulfill.quantity / demandQty : 0.0
        firstPriorityDemand.leftRatio -= ratio
    }

    supply.leftQuantity -= fulfill.quantity
    supply.leftQuantity.roundHalfDown(7)

    if (!needUpdateSupplies.contain(supply)) {
        needUpdateSupplies.add(supply)
    }

    fulfill.availableDate = supply.availableReleaseDate
    fulfill.availableTime = supply.availableReleaseTime
}
```

## 3. 删除非法冻结分配

循环全部结束后：

```java
if (!invalidFrozenFulfills.isEmpty()) {
    frozenFulfills.removeIf{ invalidFrozenFulfills.contain(it) }
    dh.deleteBatch(invalidFrozenFulfills)
}
```

然后再执行原来的更新：

```java
dh.updateBatch(needUpdateDemands)
dh.updateBatch(needUpdateSupplies)
dh.updateBatch(frozenFulfills)
```

这样被禁用的冻结分配会删除，需求不会增加已满足量，后续普通分配可以重新选择其他库位。
