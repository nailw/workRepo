## 一、最初的禅道任务：物料车间隔离

任务背景是车间排程模型通过BOM获取半成品订单物料组件，但部分半成品核料时不能使用指定库存地点。

ERP库存地点示例：

| 库位 | 名称 | 状态 | 激活/高活性 |
|---|---|---|---|
| 3410 | 原料药三车间材料库 | 有效 | 否 |
| 3411 | 原料药一车间材料库 | 有效 | 是 |
| 3412 | 原料药二车间材料库 | 有效 | 否 |

解决方案是在KTProductRouting增加文本字段NoLocationId，维护产品禁用的库存地点代码；核料供应查询时，将供应的sapStockingPointId与禁用列表比较，匹配则删除。产品12020228禁用3411后只能使用3410、3412；未维护禁用库位的产品可以使用全部有效库位。

## 二、DO平台语法结论

- 方法签名、返回类型和参数由平台界面配置，编辑区填写方法体；
- 支持var、new List<T>()、new Map()、Group<T>()、context.KT<T>()；
- 空值使用?.和??；
- 集合常用filter、map、groupBy、removeIf、firstOrNull、find、sortedBy、distinctBy；
- 集合包含使用contain；
- 支持for(Int i=1;i<13;i+=1)；
- 日期支持month、year、dayOfMonth、minusMonths、minusDays、toDate；
- 支持元组(value1,value2)，通过tuple[0]和tuple[1]读取；
- 模型字段必须按真实名称使用，例如leftQuantityVritual不能改成leftQuantityVirtual；
- 日志在已有Utils代码中使用log.info(String)，但不同模型上下文可能存在名称解析差异，需参考当前类已成功编译的写法；
- 复杂排查应一次性打印过滤前后数量、分类条件、物料代码、来源和剩余量；
- 代码减少无意义换行，便于DO编辑器复制。

## 三、核料总体流程

核料主链路：

~~~text
核料开关判断
→ 创建步骤投料和产出
→ 同步库存、采购、原料药、其他场景生产计划、本车间生产计划供应
→ 根据InputOnStep生成MaterialDemand
→ 删除未冻结分配并重占冻结分配
→ 查询可用供应
→ 按第一优先级、替代组和配套组分配
→ 生成MaterialFulfill
→ 更新需求和供应剩余量
→ 计算PieceStep最早可用时间
→ 发送DM
~~~

主要方法包括synInputAndOutputOnStep、synMaterialSupply、synMaterialDemand、assignSuppliesToMaterialDemands、reOccupyFrozenMaterialFulfill、updateFulfillTimeToPieceStepTime和sendFulfillsToDM。

历史性能：同步供应和需求约35秒优化到17秒；物料分配约361秒优化到87秒；最早可用时间计算约1392秒，是主要瓶颈；DM发送约15秒。

## 四、需求、供应和分配字段

MaterialDemand核心字段：

- demandQuantity：原始需求量；
- fulfillQuantity：实际满足量；
- leftRatio：第一优先级或替代组剩余比例；
- isFirstPriority：是否第一优先级；
- isFirstReplace：是否替代组第一顺位；
- replaceGroupId：替代组；
- replacePriority：替代顺序；
- inputFactor：投料系数。

需求量生成逻辑：

~~~java
obj.initDemandQuantity=input.isFirstPriority?input.inputFactor*pieceStep.quantity:0.0
obj.demandQuantity=obj.initDemandQuantity/scrapRate
obj.leftRatio=input.isFirstPriority?1.0:0.0
~~~

MaterialSupply核心字段：

- leftQuantity：普通供应剩余量；
- leftQuantityVritual：半成品/生产计划供应虚拟剩余量；
- availableReleaseTime和availableReleaseDate；
- dataSource；
- uniqueId和nr。

生产计划供应包括OtherSceneProductionPlan和ThisSceneProductionPlan。

## 五、本车间生产供应不分配的排查结论

问题对象：

- 成品：09020003；
- 半成品：12030001；
- 供应：BCP1-108-1等；
- 需求量：240000。

已经排除：

- 订单暂停；
- 替代有效期提前返回；
- 需求量为0；
- leftRatio为0；
- 物料代码匹配失败；
- Piece或Peizhi步骤不存在；
- 生产月份过滤；
- forbiddenMix；
- 设备组禁用物料。

日志顺序：

~~~text
初始供应 count=5
设备组过滤后 count=5
数量过滤前
数量过滤后 count=0
~~~

因此供应是在数量过滤阶段被全部删除的。

数量过滤逻辑：

~~~java
if(isGuojiChengpingUseBomBanchengping){
    outputSupplies.removeIf{MaterialSupply.getGuojiBanchengpingAvailableQty(it)<=Constant.EPS}
}else if(isDomesticSemiDemand){
    outputSupplies.removeIf{it.leftQuantityVritual<=Constant.EPS}
}else{
    outputSupplies.removeIf{it.leftQuantity<leastQty}
}
~~~

当时两个分类都是false：

~~~text
isGuojiChengpingUseBomBanchengping=false
isDomesticSemiDemand=false
~~~

所以生产计划半成品被按普通物料读取leftQuantity，而不是leftQuantityVritual，最终全部删除，分配方法收到空供应集合。

## 六、国内/国际订单分类

国际成品判断要求：

~~~java
(it?.productType1??"")=="C"&&(it?.customType??"")=="国际订单"
~~~

09020003当时为productType1=C、customType为空、productFrom108=true，因此不属于国际成品集合。

国内半成品判断要求：

~~~java
var banchengpingIds=context.KT<KTProductRouting>().filter{
    (it?.productType1??"")=="B"&&it.customType=="国内订单"
}.map{it.productId}
~~~

12030001当时为productType1=B、customType为空、productFrom108=true，因此也不属于国内半成品集合。

若业务是国内订单，建议维护：

~~~text
09020003.customType=国内订单
12030001.productType1=B
12030001.customType=国内订单
~~~

若业务是国际订单，建议维护：

~~~text
09020003.productType1=C
09020003.customType=国际订单
~~~

并确认bomBanchengpingIds包含12030001。108线产品字段当前不会自动代表国内或国际订单。

长期建议统一按照“需求成品订单类型+投料是否属于BOM半成品”判断，不要只依赖半成品自身客户类型。

## 七、替代料满足规则

主物料通常为：

~~~text
isFirstPriority=true
demandQuantity=正常需求量
leftRatio=1.0
~~~

替代料通常为：

~~~text
isFirstPriority=false
demandQuantity=0
leftRatio=0
~~~

因此使用替代料时：

- 主物料demandQuantity不会改成0；
- 替代料原始demandQuantity通常为0；
- 替代料fulfillQuantity记录实际满足量；
- 替代组整体满足情况通过第一顺位需求leftRatio表示。

满足关系通过：

~~~java
MaterialFulfill.createFulfillByDemandAndSupply(demand,supply,qty)
~~~

生成，其中demand是实际被使用的需求，所以MaterialFulfill.quantity是实际分配数量，demandUniqueId指向实际需求，materialId通常是实际使用的替代料代码，replaceGroupId和replacePriority保留替代信息。

识别是否使用替代料：

~~~java
var actualDemand=MaterialDemand.findByUniqueId(context,fulfill.demandUniqueId)
var firstDemand=MaterialDemand.findFirstReplacePriorityByMaterialDemand(context,actualDemand)
var isReplace=actualDemand!=null&&firstDemand!=null&&actualDemand.uniqueId!=firstDemand.uniqueId
~~~

页面建议同时展示主需求物料代码、实际满足物料代码、是否替代料、替代组、替代顺位、主需求量、本次满足量和实际供应编号。

截图含义：

~~~text
主料08020144：需求183.6，满足0
替代料08020003：需求0，满足183.6
实际绑定供应物料：08020003
~~~

表示替代料08020003实际满足了主料08020144的183.6需求。

## 八、后续发现的代码风险

1. 国内半成品分配代码中，实际分配量为qty1，却直接将remainQty设置为0，可能只使用第一条供应就结束，需要确认是否符合整批通过业务。
2. 国内与国际分类标准不一致，同一半成品同时服务不同订单时可能产生歧义。
3. querySemiProductIdsByBom中的半成品推导条件需要用真实BOM数据验证，不能未经回归测试直接改动。
4. 只修改查询阶段的leftQuantity取值不够，正式分配和扣减逻辑也必须使用一致的可用量规则。
