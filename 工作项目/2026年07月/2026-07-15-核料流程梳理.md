# 核料流程梳理与禁用库位需求分析

- 日期：2026-07-15
- 主题：车间排程系统核料流程、平台语法、产品禁用库存库位需求
- 记录来源：ChatGPT 对话中用户提供的代码与模型截图
- 当前状态：已完成整体流程梳理；禁用库位的新分配过滤位置已定位，尚未实施代码修改

## 1. 业务背景

车间排程模型通过 BOM 获取半成品订单的物料组件。部分半成品在核料时不能使用个别库存代码的物料。

目标方案：

1. 在基础数据 `KTProductRouting（产品路径）` 中新增禁用库位字段 `NoLocationId`（最终大小写以模型实际字段为准）。
2. 按成品 `productId` 维护其禁用库存代码。
3. 核料时，根据需求对应的成品代码读取禁用库位；当库存供应的 `sapStockingPointId` 命中禁用列表时，该供应不可参与该需求的分配。

示例：

| 产品代码 | 禁用库位 | 可用库位 |
|---|---|---|
| 12020228 | 3411 | 3410、3412 |
| 12020071 | 空 | 3410、3411、3412 |

## 2. 平台语法特征

代码不是标准 Java、Groovy 或 Kotlin，而是一套运行于 Java/JVM 业务模型体系上的脚本 DSL。

### 2.1 方法定义

方法名、所属类、返回类型、参数和描述在平台界面右侧配置，编辑器中只写方法体。例如界面组合出的入口为：

```java
static void checkInputMaterialForClient(Context context)
```

方法体：

```java
var materialManager = MaterialManager.getInstance(context)
materialManager.checkInputMaterial()
```

### 2.2 常用语法

```java
var value = expression
var list = new List<MaterialSupply>()
var map = new Map()
```

支持安全访问和空值默认：

```java
product?.detectDuration ?? Duration.ZERO
this.owner.sceneParameter?.orderAndProcessPattern ?? ""
```

集合操作包括：

```java
forEach
filter
map
find
groupBy
distinctBy
sortedBy
firstOrNull
lastOrNull
removeIf
contain
```

场景内存对象使用：

```java
Group<PieceStep>()
Group<WorkOrder>()
indexing(SomeClass.someIndex, key1, key2)
```

基础数据使用：

```java
context().KT<KTProductRouting>().filter{ it.productId == productId }
```

数据库对象使用 QueryDSL 风格的 `Q` 类：

```java

var qDemand = QMaterialDemand.qmaterialdemand
```

代码在多个 `forEach` 闭包中使用 `return` 跳过当前元素。根据现有写法推断，平台将闭包内的 `return` 当作结束当前闭包，近似 `continue`；这一点仍应通过平台文档或小测试确认。

## 3. 核料核心对象

| 对象 | 作用 |
|---|---|
| `WorkOrder` | 工单，保存产品、BOM版本等信息 |
| `Piece` | 工单拆分后的排程件/批次 |
| `WorkOrderStep` | 工单工艺步骤 |
| `PieceStep` | 某个 Piece 对应的实际排程步骤 |
| `InputOnStep` | 工序投入定义，来自 BOM 或工序 BOM |
| `OutputOnStep` | 工序产出定义 |
| `MaterialDemand` | 根据步骤投入和生产数量生成的实际物料需求 |
| `MaterialSupply` | 库存、采购、生产计划等统一供应节点 |
| `MaterialFulfill` | 某供应向某需求分配的数量关系 |

关键字段关系：

```text
MaterialDemand.productId   = 当前生产的成品代码
MaterialDemand.materialId  = BOM要求的原料/半成品代码
MaterialSupply.productId   = 供应节点所供应的物料代码
MaterialSupply.sapStockingPointId = 库存供应所在的ERP库位
```

因此供需首先通过下面的关系匹配：

```text
MaterialSupply.productId == MaterialDemand.materialId
```

禁用库位则使用：

```text
MaterialDemand.productId -> KTProductRouting.NoLocationId
MaterialSupply.sapStockingPointId -> 当前库存供应的库位
```

## 4. 核料总入口

客户端入口：

```text
PiecePlanner.checkInputMaterialForClient(context)
└─ MaterialManager.getInstance(context).checkInputMaterial()
```

主流程：

```text
检查核料开关
  ↓
同步需求和供应 synDemandAndSupply
  ↓
删除未冻结分配并重占冻结分配
  ↓
物料分配 assignSuppliesToMaterialDemands
  ↓
计算物料最早可用时间并更新步骤时间
  ↓
将物料分配结果发送给 DM
```

主方法中的具体调用顺序：

```java
if (!this.enabledCheckInputMaterial()) {
    return
}

this.synDemandAndSupply()
this.clearUnfrozenAndReOccupyFrozenMaterialFulfills()

var pieceSteps = Group<PieceStep>().toList()
this.assignSuppliesToMaterialDemands()

this.updateFulfillTimeToPieceStepTime(pieceSteps)
this.sendFulfillsToDM()
```

## 5. 同步需求和供应

`synDemandAndSupply()` 的顺序：

```text
synInputAndOutputOnStep
  ↓
synMaterialSupply
  ↓
synMaterialSupplyAvailableQuantity
  ↓
synMaterialDemand
```

其中：

1. 先生成步骤投入和产出；
2. 再汇总各种供应；
3. 扣除其他车间占用；
4. 最后根据步骤投入生成实际需求。

## 6. 步骤投入和产出生成

系统根据场景参数选择 BOM 模式：

```java
var pattern = this.owner.sceneParameter?.orderAndProcessPattern ?? ""

if (pattern == ModelConstants.Pattern23) {
    this.synInputAndOutOnStepByOperationBOM()
} else {
    this.synInputAndOutOnStepByBOM()
}
```

### 6.1 Pattern23：工序 BOM

数据通过 DM 接口 `sendI23_ImportBomAndOperationToPP` 获取。

主要处理：

1. 将所有 `InputOnStep`、`OutputOnStep` 标记为脏数据；
2. 按工单查找工序 BOM；
3. 调用 `createInputByOperationBOM` 创建各步骤投入；
4. 在工单最后步骤创建 `OutputOnStep`；
5. 删除本次没有重新生成的旧投入和产出；
6. 批量加入本次新建对象。

可疑点：数据按 `queryWorkOrderId` 分组，但使用 `workOrder.queryOperationBomId` 查找，需确认两个字段是否设计为相同值。

### 6.2 普通 BOM

处理分为三部分。

#### 特殊工序投入

查询 `KTInputConstraint`，把有特殊步骤约束的物料放到指定步骤：

```java
var ktRows = KT<KTInputConstraint>().select(
    it.productId == productId,
    it.stepSequenceNr != ""
)
```

同时通过 `canUseBomByInputProductId` 判断当前订单是否允许使用该 BOM 物料。

#### 普通 BOM 投入

没有特殊约束的 BOM 物料在每个 `Piece` 的第一步骤创建 `InputOnStep`。

创建时先查询已经位于场景 `Group` 中的对象，再查询本轮刚创建但尚未加入 `Group` 的 `toAddInput`，避免重复创建。

#### 产出

对每个订单下的每个 `Piece`，在选中步骤中序号最大的末端步骤创建 `OutputOnStep`。

潜在问题：代码按 `productId + bomVer` 分组，但查询 BOM 时只传 `productId`，可能没有真正区分 BOM 版本；字符串直接拼接也存在键碰撞风险。

## 7. PieceStep 生成

系统遍历 `Piece.workOrderSteps`，根据 `Piece` 与步骤组合查找或创建 `PieceStep`。

定制逻辑会读取 `KTProductRouting` 中的：

```text
pieceSizeQuantity1
operator
```

根据 Piece 数量以及“＞/＜”规则选择或排除自动包装步骤 `Zidongbaozhuang`，并维护 `piece.selectedWorkOrderSteps`。

最后删除未重新生成的旧 `PieceStep`，加入新步骤，并同步设备可用性。

## 8. MaterialDemand 生成

`synMaterialDemand()` 调用：

```java
this.synInputStepToMaterialDemand()
```

生成方法先删除旧需求，再遍历 `Group<PieceStep>()`：

```java
Group<PieceStep>().forEach{ pieceStep ->
    if (pieceStep.cannotCheckInputReason != "") {
        return
    }

    pieceStep.workOrderStep?.inputsOnStep?.forEach{ input ->
        var obj = new MaterialDemand()
        MaterialDemand.updateAttributeByPieceStepAndInput(obj, pieceStep, input)
        toAdd.add(obj)
    }
}
```

需求数量计算：

```java
obj.initDemandQuantity = input.isFirstPriority
    ? input.inputFactor * pieceStep.quantity
    : 0.0

var scrapRate = obj.scrapRate ?? 1.0
if (scrapRate <= 0.0) {
    scrapRate = 1.0
}

obj.demandQuantity = obj.initDemandQuantity / scrapRate
obj.scrapQuantity = obj.demandQuantity - obj.initDemandQuantity
```

需求唯一标识：

```java
obj.uniqueId = pieceStep.pieceStepId
    + "|" + input.materialId
    + "|" + input.lineNr
    + "|" + input.replaceGroupId
```

第一优先级需求初始化：

```java
obj.leftRatio = input.isFirstPriority ? 1.0 : 0.0
```

风险：`dh.dslDelete(qdemand).execute()` 没有显式场景条件。如果各场景不是物理隔离表，需要确认是否会删除其他场景需求。

## 9. MaterialSupply 生成

`synMaterialSupply()` 先删除全部旧供应，然后依次生成：

```text
ERP库存
分批到货原料药
采购订单
采购计划/采购申请
其他场景生产计划
本车间生产计划产出
```

### 9.1 ERP库存

通过 BOM 推导半成品物料集合。

半成品：

- 只使用库存 `quantity`；
- 不拆待检量；
- 使用半成品入库/生产日期；
- 可能设置虚拟供应量。

普通原辅包拆成：

1. 可用库存 `quantity`；
2. 待检库存 `detectQuantity`。

待检库存的释放时间为：

```text
入库时间 + 产品检验周期 detectDuration
```

库存源对象 `InventoryIn` 已有：

```text
sapStockingPointId
sapStockingPointName
```

但当前 `MaterialSupply` 模型及三个库存转换方法都没有复制该字段，因此现有分配阶段无法知道供应来自哪个库位。

### 9.2 其他供应

- 原料药来自 DM 接口 `sendI36_ImportSupplyPlanToPP`；
- 采购订单根据检验周期计算释放时间；
- 采购计划同样根据检验周期计算释放时间；
- 其他场景生产计划来自 `sendPP_ProductionPlanToPP`；
- 本车间供应根据 `PieceStep.outputsOnStep` 生成。

## 10. 删除旧分配与冻结重占

系统删除所有未冻结的 `MaterialFulfill`：

```java
qfulfill.frozenFlag.eq(false) || qfulfill.frozenFlag.isNull()
```

然后重新占用冻结分配：

1. 按 `demandUniqueId` 分组；
2. 找到对应需求；
3. 更新需求 `fulfillQuantity`；
4. 更新第一优先级需求的 `leftRatio`；
5. 根据 `supplyUniqueId` 找到供应；
6. 扣减供应 `leftQuantity`；
7. 用当前供应更新 Fulfill 的可用日期和时间。

现有风险：需求满足量在确认供应存在之前就被增加。如果冻结分配对应供应已经不存在，会出现需求显示已满足、但没有实际供应扣减的状态。

此外，冻结重占不会调用新分配的供应过滤逻辑。若产品后来新增禁用库位，旧冻结分配仍可能继续占用该库位，需要单独确定业务规则。

## 11. 新物料分配总流程

系统一次性加载全部需求和供应：

```java
var supplyNode = MaterialSupply.queryAllData(context())
var demandNode = MaterialDemand.queryAllData(context())
```

只先取第一优先级需求：

```java
var firstDemands = MaterialDemand.queryFirstPriorityFromInputDemand(demandNode)
```

然后进入：

```java
MaterialDemand.assignSuppliesToFirstDemands(
    context(),
    firstDemands,
    supplyNode,
    demandNode
)
```

## 12. 配套组预处理和需求排序

场景参数控制配套组能否混用：

```java
var canMixConsumeSet = SceneParameter.getInstance(context)?.canMixConsumeSet ?? false
```

有 `consumeSet` 的需求按 `pieceStepId + consumeSet` 去重，只保留一条代表需求用于排序；没有配套组的需求全部保留。

之后通过 `sortedByDueDateAndPieceStepId` 排序。

风险：直接拼接 `pieceStepId + consumeSet` 可能发生键碰撞，建议使用分隔符：

```java
pieceStepId + "|" + consumeSet
```

## 13. 普通需求与替代料分配

无配套组需求进入：

```java
assignSuppliesToFirstReplaceDemand(...)
```

流程：

1. 查询第一替代顺位需求可用的供应；
2. 调用 `assignAvailableSuppliesToDemand` 正式分配；
3. 累加满足比例和 `MaterialFulfill`；
4. 若已完全满足则返回；
5. 若存在 `replaceGroupId`，按 `replacePriority` 查询下一替代顺位；
6. 循环使用替代料，直到满足或没有更多替代需求。

第一顺位查询：

```java
MaterialSupply.queryAvailableFromInputSupplies(
    supplyNode,
    firstDemand,
    Constant.EPS,
    context
)
```

替代顺位当前传入：

```java
MaterialSupply.queryAvailableFromInputSupplies(
    supplyNode,
    demand,
    firstDemand.demandQuantity,
    context
)
```

由于查询内部对普通供应执行 `leftQuantity < leastQty` 过滤，这会要求单条替代供应至少覆盖完整需求。即使允许混批，多批合计足量也可能被全部排除。代码注释显示这是一次定制修改，需要确认原业务目的。

## 14. 配套组分配

有 `consumeSet` 的需求进入：

```java
assignSuppliesToFirstDemandSet(...)
```

首先尝试当前第一配套组；不足时，通过相同替代组寻找下一配套组优先级并依次尝试。

`assignSuppliesToDemandSet()` 的逻辑：

1. 只选择配套组内 `isFirstReplace` 的主需求；
2. 对每个主需求进行临时分配；
3. 取所有主需求满足率的最小值作为配套组瓶颈；
4. 如果不允许与其他配套组混用且不能完全满足，则本组满足率置零；
5. 如果可以分配，按瓶颈满足率对每个主需求正式分配；
6. 若使用的是替代配套组，扣减第一配套组的剩余比例。

临时分配与正式分配最终都调用：

```text
assignSuppliesToFirstReplaceDemand
└─ queryAvailableFromInputSupplies
```

因此普通需求、替代料和配套组共享同一个供应过滤入口。

## 15. 公共供应过滤

`MaterialSupply.queryAvailableFromInputSupplies()` 首先按物料代码缩小供应范围：

```java
var supplyMap = inputSupplies.groupBy{ it.productId }
var outputSupplies = supplyMap.getOrDefault(
    materialDemand.materialId,
    new List<MaterialSupply>()
)
```

随后依次执行：

1. 已排产半成品按设备组禁用物料过滤；
2. 判断国内半成品或国际成品使用 BOM 半成品；
3. 按普通、虚拟或国际半成品可用量过滤；
4. 国际成品 BOM 半成品按生产月份过滤；
5. 不可混批约束；
6. 物料有效期约束；
7. 指定供应商约束；
8. 指定上游车间约束。

最终返回候选供应列表。

现有设备组禁用物料代码把 `NoMaterialsID` 当作单个字符串进行相等比较。如果该字段可保存多个物料代码，需要先用 `Utils.splitStringToStringList` 拆分。

## 16. 禁用库位需求的正确落点

禁用库位不能在 `synInventoryToMaterialSupply()` 阶段全局删除库存，因为同一库位供应对产品 A 可能禁用，对产品 B 仍可用。

正确位置是：

```java
MaterialSupply.queryAvailableFromInputSupplies()
```

原因：

- 方法同时持有当前 `MaterialDemand`、候选 `MaterialSupply` 和 `context`；
- 普通需求会经过该方法；
- 替代料会经过该方法；
- 配套组临时分配和正式分配也会经过该方法。

需要的最小修改：

1. `KTProductRouting` 新增 `NoLocationId String`；
2. `MaterialSupply` 新增 `sapStockingPointId String`；
3. 可用库存、待检库存、半成品库存转换时复制 `inventory.sapStockingPointId`；
4. 在公共供应过滤中，读取 `materialDemand.productId` 对应的禁用库位；
5. 对库存来源供应执行库位排除；
6. 采购和生产计划等没有库存库位的供应不受影响。

判断示意：

```java
var noLocationIds = new List<String>()

var productRoutings = conetxt.KT<KTProductRouting>()
    .filter{ it.productId == materialDemand.productId }

productRoutings.forEach{ routing ->
    var locationIds = Utils.splitStringToStringList(
        routing.NoLocationId ?? ""
    )
    noLocationIds.addAll(locationIds)
}

if (!noLocationIds.isEmpty()) {
    outputSupplies.removeIf{
        MaterialSupply.isInventorySupply(it) &&
        it.sapStockingPointId != "" &&
        noLocationIds.contain(it.sapStockingPointId)
    }
}
```

字段大小写必须与模型实际定义完全一致。

## 17. 性能建议

公共供应查询会被普通需求、替代料、配套组临时模拟和正式分配多次调用。若每次都扫描 `KTProductRouting`，可能增加耗时。

可选优化：

1. 给 `MaterialDemand` 增加 `noLocationId String`；
2. 生成需求时一次性读取并写入产品禁用库位；
3. 分配时直接从需求拆分禁用库位，不再查询 KT。

另一方案是在分配开始前构建 `Map<产品代码, 禁用库位集合>`，但需要修改多层方法签名以传递该 Map。

## 18. 已识别的其他风险

1. 工序 BOM 分组键和查找键疑似不一致；
2. 普通 BOM 按产品和版本分组，但实际查询可能忽略 BOM 版本；
3. `productId + bomVer`、`pieceStepId + consumeSet` 直接拼接存在碰撞风险；
4. 冻结分配对应供应不存在时，需求仍被计入满足量；
5. 冻结分配不会经过禁用库位过滤；
6. 原料药和其他场景生产计划未看到统一的零/负数量过滤；
7. `supply.leftQuantity.roundHalfDown(7)` 是否原地修改需要确认；
8. 替代料查询传入完整需求量，可能错误排除可组合的多批供应；
9. 设备组 `NoMaterialsID` 如果是多值字段，当前字符串相等判断可能失效；
10. 多条 `KTProductRouting` 对应同一产品时，需要确定禁用库位是合并、取当前路径还是只取第一条。

## 19. 尚待业务确认

1. `NoLocationId` 是否允许维护多个库位，分隔符是否统一使用逗号；
2. 同一产品存在多条 `KTProductRouting` 时的取值规则；
3. 已冻结且使用了现已禁用库位的分配，是继续保留、解除冻结，还是报错；
4. 禁用规则是否覆盖普通原辅包库存、待检库存和半成品库存三类；
5. 是否需要把实际库位复制到 `MaterialFulfill` 供 DM 输出和审计。

## 20. 当前结论

整体核料流程已经梳理完毕。禁用库位的新分配过滤只需放在公共方法 `MaterialSupply.queryAvailableFromInputSupplies()`，即可覆盖普通需求、替代料和配套组；但冻结分配必须另行处理。实施前还需确认第 19 节的业务规则。
